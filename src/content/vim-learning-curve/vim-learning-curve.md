## 初見の情報量が少ない

これはメリットでもありデメリットになりうるのですが、vimは初見の起動時に何をすればいいのか分からないくらいにシンプルです。

少なくとも私にとっての学習曲線の難易度は認知負荷を超えた情報が流れやすいかどうかです。
色々な機能などの事柄が線形的に並んでおり、それを順に獲得していくことが主とされているようなソフトウェアの学習曲線は緩やかであることが多い
それの例がvim

例えばgitはまだいい方だがゆるいとは言えない
それはいきなりgit status, commit, add, logなどのサブコマンドが順をほぼ持たずに並列的に流れ入ってくるため
本当にひどいのはnixOS。あり得ない量の情報が一気に流れる これと決まった線形ドキュメントがないというのもある?

chromeなどは好例でまずボタンが少なく、それに伴い機能も少ない。上のボタンでtabが増え、xボタンでtabが閉じる。上部のボタンで戻ったり特定のURLへ移動
まあwebサーフィンという一種の前提があってのものではあるが、gitの構造よりはシンプルでプリミティブであると言えよう。

iOSなどはそのあたりを目指しているようにも感じる。liquid glassはやりすぎだと思うが。

そしてvimもその好例に数えてもいいのではないかと考えることができる。端切れが悪いのはいささか前提を必要とするので学習曲線は緩やかだが長いという表現が適切なのではという懸念があるため。

vimの比較対象としてVS Codeをあげる
VS Codeは現代的でプラグインシステムとlspという革新的な技術によってエディターとIDEのような言語仕様との分離と連携を可能にした素晴らしいエディターだが、近年はいささかUIの複雑化を感じる。
それでもよく収まっている方ではあるだろうが。 ~~ MS製とは思えないほどに ~~

VS Codeは起動すると多くのボタンが待ち構える。これが一気に流れてくる。GUIは便利だがそのボタンに順序性を持たせるには技術力がいる。デザイン力?
実際のところVS Codeは初見起動ではアナウンスがあり、基本機能のチェックシートをステップごとに出してくれるが、多くのプログラマは機械の言うことを聞かない上にせっかちなのでdone done進んでしまう。

その点vimはユーザが何をすればいいのか最小限しか教えない。いや正確にはhelpの表示とquitの仕方は教えてくれるのだけれども
まずvimについて何も知らないならば、文字の入力ができないだろう。ここで詳細を語るのは脇にそれすぎるのでやらないがvimにはいくつかのモードがあり、`i`などの特定のキーを押さないと文字を入力する`insert-mode`に移行できない。

そういうものを教えてくれるチュートリアルは(おそらく)ないし、一般的ではない。そのためエディターとしてプリミティブな操作でさえ、調べる必要がある。
これが最大のメリットでありデメリットであると言える。
(いささか)
多くの操作を調べながら行う必要があるということは、それだけvimについて知った状態で進めるということ。少なくとも理解しようと思ったことを理解しながらゆっくり学習することができる。
これは一種、プログラミング言語の学習にも似ている気がして、データを保持したい -> 変数, データをまとめたい -> structure のように動機から理解した機能というのは頭に残りやすい。


ただこれらはあくまで現代に生まれてそれなりにPCという環境に育った私のバイアスでもって成立している話なので個別具体の事例に関してはそれぞれに難易度の差があるとは思う。
